# Как участвовать в проекте

Спасибо, что интересуешься улучшением этого проекта! 🙌

## 📋 Типы вкладов

Ты можешь помочь несколькими способами:

### 🐛 Исправление ошибок
- Ошибки в коде примеров
- Опечатки в документации
- Неправильные объяснения концепций

### 📚 Улучшение документации
- Более ясные объяснения
- Лучше структурированные примеры
- Дополнительные примеры использования

### ✨ Новые задания
- Новые практические задачи
- Более сложные примеры
- Задачи для закрепления материала

### 🎯 Новый контент
- Новые блоки обучения
- Новые темы и модули
- Проектные работы

## 🔄 Процесс внесения изменений

### 1. Fork репозитория
Нажми кнопку "Fork" в GitHub

### 2. Создай веточку для своих изменений
```bash
git checkout -b feature/улучшение-переменных
# или
git checkout -b fix/опечатка-в-readme
```

**Именование веток:**
- `feature/` — новые задания, материал
- `fix/` — исправления ошибок
- `docs/` — улучшение документации
- `refactor/` — переструктурирование

### 3. Внеси изменения

Пожалуйста, следуй этим правилам:

**Для кода:**
```python
# ✅ Хорошо
def calculate_circle_area(radius):
    """Вычисляет площадь круга."""
    PI = 3.14159
    return PI * radius ** 2

# ❌ Плохо
def calc(r):
    return 3.14159 * r ** 2
```

**Для документации:**
- Используй ясный, простой язык
- Добавляй примеры где возможно
- Следуй существующему стилю

**Для заданий:**
- Задание должно включать описание и пустое место для кода
- Добавь примерное решение в комментариях или отдельно
- Укажи уровень сложности

### 4. Напиши понятный коммит

```bash
git add .
git commit -m "feat: add 5 new tasks for loop practice"
# или
git commit -m "docs: fix typo in variables explanation"
# или
git commit -m "fix: correct PI value in circle calculation example"
```

**Формат коммитов:**
```
type: краткое описание (максимум 50 символов)

Подробное описание (опционально)
- Что было добавлено
- Какие проблемы решены
```

**Типы коммитов:**
- `feat:` — новое задание или материал
- `fix:` — исправление ошибок
- `docs:` — документация
- `refactor:` — переструктурирование
- `style:` — форматирование кода
- `test:` — тесты

### 5. Запушь в свой fork
```bash
git push origin feature/улучшение-переменных
```

### 6. Создай Pull Request
1. Перейди на GitHub
2. Нажми "Compare & pull request"
3. Напиши описание: что ты изменил и почему
4. Отправь PR

## 📝 Описание Pull Request

Используй этот шаблон:

```markdown
## Описание
Краткое описание того, что ты сделал.

## Тип изменения
- [ ] 🐛 Исправление ошибки
- [ ] ✨ Новое задание
- [ ] 📚 Улучшение документации
- [ ] 🔄 Переструктурирование

## Изменения
- Что было добавлено/исправлено
- Почему это нужно
- Примеры

## Связанные issues
Закрывает #123 (если применимо)

## Как тестировать
Инструкции для проверки изменений:
```bash
python path/to/file.py
```
```

## ✅ Чек-лист перед PR

- [ ] Код запускается без ошибок
- [ ] Нет опечаток в документации
- [ ] Код следует PEP 8 стилю
- [ ] Примеры имеют полезные комментарии
- [ ] Задания имеют понятные инструкции
- [ ] Файл `.gitignore` правильно работает

## 🎨 Стиль кода

Мы следуем [PEP 8](https://pep8.org/) — стандарту Python.

### Основные правила:

```python
# ✅ Правильно (snake_case)
student_name = "Alice"
max_attempts = 5

# ❌ Неправильно (camelCase)
studentName = "Alice"
maxAttempts = 5

# ✅ Правильно (константы CAPS_LOCK)
PI = 3.14159
DATABASE_URL = "localhost"

# ✅ Правильно (4 пробела на отступ)
if condition:
    do_something()

# ✅ Правильно (пробелы вокруг операторов)
total = price * quantity + tax
```

## 📚 Структура файлов

```
01_junior/
├── 2.1_fundamentals/
│   ├── 03_variables_demo.py    # Теория и примеры
│   ├── 03_variables_task.py    # Практика с пустыми местами
│   └── README.md               # Описание этого блока
└── README.md                   # Описание уровня junior
```

**Для новых блоков:**
- `XX_topic_demo.py` — примеры и объяснения
- `XX_topic_task.py` — практические задания
- Все задания должны иметь пустые места `# ТВ КОД ЗДЕСЬ:`

## 🔍 Что мы ищем

✅ **Хорошие PR:**
- Ясные объяснения
- Полезные примеры
- Хорошо структурированные задания
- Исправления реальных ошибок

❌ **Плохие PR:**
- Случайные изменения в разных файлах
- Значительное переписывание существующего контента
- Добавление неправильных примеров
- Спам или реклама

## 💬 Обсуждение

Если ты не уверен, нужно ли твоё изменение, создай [Issue](../../issues) сначала!

Это поможет нам обсудить:
- Нужно ли это изменение?
- Как лучше его реализовать?
- Вписывается ли оно в концепцию проекта?
